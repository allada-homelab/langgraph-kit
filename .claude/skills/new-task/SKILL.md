# new-task

Drives one issue from open to closed **in the current Claude Code session**. Does not dispatch subagents. Does not hand off. Every stage below runs in this same session — when the skill says "follow `superpowers:feature-dev`", it means load that skill's instructions and run them here, not spawn an agent.

Parallelism is external: run this skill in multiple sessions, each rooted in a different `git worktree` (see [`worktree-for`](../worktree-for/SKILL.md)).

**Invocation:** `/new-task <issue-number>` or mentioned as "work on issue N".

## Setup (shell vars the whole pipeline uses)

Derive these once at the top, then reuse. Never hardcode the repo name.

```bash
N="$1"                                                        # issue number
REPO="$(gh repo view --json nameWithOwner -q .nameWithOwner)" # e.g. allada-homelab/langgraph-kit
GH_MAJOR_MINOR="$(gh --version | head -1 | awk '{print $3}' | cut -d. -f1-2)"
# SLUG derivation happens in stage 2 after we have the title.
```

## Pipeline

Each stage is sequential. Surface to the user on any stage-level failure not covered by an explicit retry rule.

### Stage 0. Resume check

Before anything else, detect in-progress state for this issue and skip forward:

```bash
EXISTING_BRANCH="$(git for-each-ref --format='%(refname:short)' refs/heads/ | grep -E "^feature/${N}-" | head -1)"
```

- No existing branch → run stage 1.
- Branch exists, currently checked out, matches this issue → figure out resume point:
  - No commits beyond `origin/main` → resume at stage 3 (implementation).
  - Commits exist, no open PR → resume at stage 5 (PR creation).
  - Open PR exists → resume at stage 6 (CI loop).
- Branch exists but **not** checked out → abort. Ask the user to check it out or delete it.

### Stage 1. Pre-flight

```bash
# Issue must exist and be open. Capture title + body for later stages.
gh api "repos/$REPO/issues/$N" --jq '{title, body, state, labels: [.labels[].name]}'
# If .state != "open" → abort.

# Working tree must be clean.
test -z "$(git status --porcelain)" || {
  echo "dirty working tree — run \`git stash -u -m 'parking pre-new-task'\` and retry"; exit 1; }

# Scan the issue body for explicit blockers: `depends on #X` / `blocked by #X`.
# If any listed issue is still open, abort with the list.
```

**PR overlap (advisory, not blocking):**

Rather than trying to auto-detect collisions from the issue body, show the user what's in flight and let them confirm:

```bash
gh pr list --repo "$REPO" --state open --json number,title,headRefName,files \
  --jq '.[] | {pr: .number, title, branch: .headRefName, paths: [.files[].path] | unique[:20]}'
```

Print that summary and **ask the user** whether to proceed. This is honest — path collisions only become real during implementation.

**gh version check:** if `GH_MAJOR_MINOR` < 2.7, note that stage 6 will fall back to `gh run view --log | tail -500` instead of `--log-failed`. Non-blocking; informational.

### Stage 2. Branch setup + mark "In progress"

```bash
git fetch origin

# SLUG: title lowercased, `[N.M] ` prefix stripped, non-ASCII removed, spaces→dashes, ≤50 chars.
SLUG="<computed from title>"
BRANCH="feature/${N}-${SLUG}"

git checkout -b "$BRANCH" origin/main
```

Move the Project #1 card for this issue to **Status = In progress** so other sessions see it's taken. Dynamic lookup (no hardcoded IDs):

```bash
# 1. Find the project item ID for this issue.
ITEM_ID="$(gh api graphql -f query='
  query($repo_owner: String!, $repo_name: String!, $issue: Int!) {
    repository(owner: $repo_owner, name: $repo_name) {
      issue(number: $issue) { projectItems(first: 10) { nodes { id project { number } } } }
    }
  }' -F repo_owner=${REPO%/*} -F repo_name=${REPO#*/} -F issue=$N \
   --jq '.data.repository.issue.projectItems.nodes[] | select(.project.number == 1) | .id')"

# 2. Look up project id, Status field id, and the "In progress" option id (or cache statically).
#    Hardcoding these is acceptable because they are stable per project;
#    query dynamically if the skill ever ships beyond this repo.
# 3. updateProjectV2ItemFieldValue(...singleSelectOptionId: <in-progress>)
```

Non-fatal if the board update fails — log a warning and continue.

### Stage 3. Implementation

Load the `superpowers:feature-dev` skill and follow its steps **in this session**. Seed with:

- The issue title + body fetched in stage 1.
- Scope hints: top-level directories mentioned in the issue, plus `tests/` for any of them.
- Hard scope: never hand-edit `testapp/`, `uv.lock`, or `checkpoints.db` (the `block-protected-paths` hook blocks them anyway). `uv.lock` being regenerated by `uv sync` / `uv lock` is **fine** — that's the tool doing its job, not a hand-edit.

If the issue body is short or ambiguous, follow `superpowers:brainstorming` first to align on scope with the user.

### Stage 4. Verification gate (capped at 3 fix iterations)

Follow the `superpowers:verification-before-completion` skill, which for this repo means running:

```bash
just lint-fix fmt typecheck test pre-commit
```

On failure, fix in-session and re-run. **Cap at 3 attempts**; beyond that, stop and surface to the user — repeated failure at this stage means the plan is wrong, not the fix.

Triage helpers:

- Coverage dropped on changed files → invoke the `coverage-floor-guardian` agent. Never lower `fail_under`.
- New pytest warning fires as error (`filterwarnings = ["error"]`) → invoke the `warning-investigator` agent.

### Stage 5. Commit, push, PR

Load and follow the `commit-commands:commit-push-pr` skill **in this session**. Explicitly pass a PR body that ends with `Fixes #<N>` — do not let auto-generation elide the trailer. The trailer is what auto-closes the sub-issue and moves the Project card to Done on merge.

Body template:

```markdown
## Summary
<2–3 bullets derived from what was implemented — not the issue body>

## Test plan
<short checklist of what was verified locally>

Fixes #<N>
```

Record the PR URL and its head commit SHA for stage 6.

### Stage 6. CI auto-fix loop (cap: 3 iterations, no busy-wait)

**Don't block the session sleeping.** Use the `ScheduleWakeup` tool (or the `loop` skill, whichever is available) to yield control between polls. User can interrupt any time.

```text
attempt = 0
while attempt < 3:
    attempt += 1
    sha = git rev-parse HEAD     # filter polls to this exact commit

    # Yield-and-re-enter loop: every ~3 min, check status.
    until all runs for `sha` on this branch have `status == "completed"`:
        runs = gh run list --branch $BRANCH --commit $sha \
                 --json databaseId,name,status,conclusion --limit 20
        if any run still pending/running:
            ScheduleWakeup(delaySeconds=180, reason="CI still running for PR #<pr>")
            # session yields; resumes when the wake-up fires

    failing = [r for r in runs if r.conclusion not in {"success","neutral","skipped"}]
    if not failing:
        print "✓ CI green on <sha>" + PR URL; EXIT LOOP

    # One flake retry per failing run, once per attempt, before spending the iteration on a fix.
    for r in failing:
        gh run rerun $r.databaseId --failed
    # Re-poll with ScheduleWakeup. If all reruns now pass, break.

    # Still failing → diagnose.
    for r in failing:
        if GH_MAJOR_MINOR >= 2.7:
            gh run view $r.databaseId --log-failed
        else:
            gh run view $r.databaseId --log | tail -500

    # Fix in-session. Re-run stage 4 (local gate) before pushing.
    # Commit + push to the same branch. New sha → next poll picks it up.

# After loop:
#   green → print "Closes #<N> on merge" + PR URL + cleanup commands (see Cleanup).
#   still red → print a terse one-screen summary (which workflows, which jobs) and stop.
#               do NOT merge; surface to user.
```

**Polling cadence:** 180s via `ScheduleWakeup`. This exceeds the 5-min prompt cache so expect some cache misses; the alternative (busy-waiting in the session) wastes user-visible time. Worth the cache miss.

### Caveats encoded for stage 6

- **Never edit a test to paper over a production bug.** If a fix requires changing an assertion's expected value, stop and ask the user. Fixing a legitimately wrong test is fine — be explicit about which you're doing.
- **`uv.lock` / `pyproject.toml` conflicts** from another PR merging first: run `git fetch origin && git rebase origin/main && uv lock && just test`. That's the fix step; counts as one iteration. (`uv lock` is a tool regenerating the file — not a hand-edit — so the protected-paths hook does not fire.)
- **Coverage floor drop only on CI** (passes locally): likely a code path not exercised under all of 3.11/3.12/3.13. Add a test that runs on all three.
- **`detect-conflicts.yml` labels the PR as conflicting**: rebase is the fix.

## When to use this skill vs. not

- **Use it** for any issue in this repo with a clear scope — typical fit: P0/P1 with a concrete body.
- **Skip it** for tiny contract-wiring issues (label fixes, trivial docs) — delegate via `@claude` on the issue instead.
- **Skip it** when the working tree is dirty — stash or commit first (stage 1 will abort).
- **Skip it** when multiple P1/P0 issues collide on the same directory under `src/langgraph_kit/` — start them sequentially.

## Parallelism

This skill is the *inside* of one parallel task. To run three in parallel:

1. In any Claude session, invoke `/worktree-for <N1>` and `/worktree-for <N2>`.
2. Open two new terminals, `cd` into each worktree, run `claude`.
3. In each session (including the original), invoke `/new-task <N>`.

No cross-session coordination — collisions surface in stage 1's open-PR summary and stage 6's CI loop (which handles rebase).

## Cleanup (printed by the success banner)

When CI turns green and the PR is ready to merge, print these for the user:

```bash
# After PR merges:
git worktree remove ../lgk-parallel/wt-<N>   # if this session was in a worktree
git branch -d feature/<N>-<slug>             # only after confirming the branch merged
```

Don't auto-run them — user confirms merge state.

## Reporting

Terse. Per-stage one-liners during execution. On success: PR URL + "Closes #N on merge." On failure: which stage, the failing step's single most useful excerpt, and the next action — not a postmortem.
